# Overview

> A Polymarket-style binary prediction market on Solana Devnet with a fully on-chain, permissionless order book.

This project is a reference implementation of a decentralized binary prediction market on **Solana Devnet**. It replaces Polymarket's centralized CLOB (Central Limit Order Book) with a **DLOB (Decentralized Limit Order Book)** architecture modeled after [Drift Protocol v2](https://docs.drift.trade/).

The on-chain program is written as a **native Solana program** — no Anchor framework — using **[Pinocchio](https://github.com/anza-xyz/pinocchio)**, the zero-dependency `no_std` Solana framework developed by Anza, and `borsh` for serialization.

---

## Why a DLOB instead of a CLOB?

Polymarket's production system keeps orders in an off-chain database operated by Polymarket Inc. While trades settle on Polygon, the operator controls which orders are accepted and matched. This project explores moving all order storage on-chain and handing matching to permissionless bots.

| Dimension | CLOB (Polymarket) | DLOB (This project) |
|-----------|-------------------|---------------------|
| Order storage | Off-chain operator database | On-chain program accounts |
| Order matching | Centralized operator | Permissionless keeper bots |
| Settlement | On-chain (Polygon Exchange contract) | On-chain (Solana program) |
| Censorship risk | Operator can censor orders | Any keeper can fill any order |
| Liveness | Depends on Polymarket infrastructure | Depends on keeper network |
| Latency | Low (centralized) | Moderate (keeper round-trip) |

---

## Why native Solana instead of Anchor?

| Dimension | Anchor | Native |
|-----------|--------|--------|
| Boilerplate | Generated by macros | Written explicitly |
| Account validation | `#[derive(Accounts)]` macro | Manual checks per instruction |
| Instruction discriminator | 8-byte sha256 hash | 1-byte enum index |
| Binary size | Larger (macro expansion) | Smaller |
| Compute budget | Higher (macro overhead) | Lower |
| Dependency surface | `anchor-lang`, `anchor-spl`, proc-macros | `pinocchio`, `borsh` only |
| IDL | Auto-generated | Hand-written TypeScript types |

The native approach keeps the program binary small, reduces CU consumption per instruction, and eliminates all Anchor toolchain dependencies. Account validation is done explicitly in each instruction handler — more verbose, but easier to audit.

## What is a binary prediction market?

A binary prediction market lets participants bet on the outcome of a yes/no question — e.g., "Will BTC be above $100k by the end of 2025?"

- Every market has exactly two outcomes: **YES** and **NO**.
- Each outcome is worth **1 USDC if it wins**, 0 USDC if it loses.
- Prices represent implied probability: an order at **0.60** (6000 basis points) implies a 60% chance of YES.
- Participants buy/sell YES or NO positions through a limit order book.
- After resolution, winners redeem their position for 1 USDC per share.

## Solana Tech Stack — May 2026

This project targets the current Solana devnet environment, which includes the following active upgrades:

### P-Token (live on devnet)

**P-Token** ([SIMD-0266](https://github.com/solana-foundation/solana-improvement-documents/pull/266), [solana.com/upgrades/p-token](https://solana.com/upgrades/p-token)) is Solana's compute-optimized replacement for the SPL Token program, developed by Anza using the **Pinocchio** framework. It was activated on devnet in April 2026 and is targeting mainnet in May 2026.

Key facts:
- **Drop-in replacement** — same program address as SPL Token, no client changes needed
- **95–98% CU reduction** on all token operations (transfer, mint, burn)
- **~10% reduction** in total Solana block space usage network-wide
- Adds `batch`, `withdraw_excess_lamports`, and `unwrap_lamports` instructions

**Impact on this project:** All USDC vault transfers (`Split`, `Merge`, `Redeem`) and outcome token mints (`TokenizePosition`) automatically use p-token's optimized rates on devnet. A `Transfer` CPI drops from 4,645 CU to just 76 CU — a 98% saving.

The program itself is also written with **Pinocchio**, achieving further CU savings in its own logic through zero-copy account access and zero heap allocations. See [Pinocchio](../program/pinocchio.md).

## Scope and limitations

This is a **devnet reference implementation**, not a production system.

- Uses a mock USDC SPL mint on devnet (not real funds).
- Market resolution is performed manually by an admin keypair (no oracle integration).
- No order expiry (GTD), no AMM fallback, and no JIT auction.
- Binary markets only — no multi-outcome support.

See [Open Questions](../resources/program-id.md#open-questions) for planned extensions.
