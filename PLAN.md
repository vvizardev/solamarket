# Polymarket Fork on Solana Devnet — DLOB Edition

## Overview

This project implements a Polymarket-style binary prediction market on **Solana devnet**, replacing Polymarket's centralized CLOB (Central Limit Order Book) with a **DLOB (Decentralized Limit Order Book)** architecture modeled after Drift Protocol v2.

The on-chain program is written as a **native Solana program** (no Anchor framework) using the `solana-program` crate directly with Borsh serialization.

### Why DLOB over CLOB?

| Dimension | CLOB (Polymarket) | DLOB (This project) |
|---|---|---|
| Order storage | Off-chain operator database | On-chain program accounts |
| Order matching | Centralized operator | Permissionless keeper bots |
| Settlement | On-chain (Polygon Exchange contract) | On-chain (Solana program) |
| Censorship risk | Operator can censor orders | Any keeper can fill any order |
| Liveness | Depends on Polymarket infra | Depends on keeper network |
| Latency | Low (centralized) | Moderate (keeper round-trip) |

### Why native over Anchor?

| Dimension | Anchor | Native |
|---|---|---|
| Boilerplate | Generated by macros | Written explicitly |
| Account validation | `#[derive(Accounts)]` macro | Manual checks per instruction |
| Instruction discriminator | 8-byte sha256 hash | 1-byte enum index |
| Binary size | Larger (macro expansion) | Smaller |
| Compute budget | Higher (macro overhead) | Lower |
| Dependency surface | `anchor-lang`, `anchor-spl`, proc macros | `solana-program`, `borsh` only |
| IDL | Auto-generated | Hand-written TypeScript types |

### Component Roles

#### 1. Native Solana Program (`program/`)

The on-chain source of truth. All state transitions require a signed Solana transaction.

**Program structure:**
- `entrypoint.rs` — `entrypoint!(process_instruction)` macro, dispatches to router
- `processor.rs` — matches `InstructionData` enum variant → calls instruction handler
- `instruction.rs` — `InstructionData` enum with Borsh-serialized payload per variant
- `state/` — Borsh-serializable account structs with fixed-size layout
- `error.rs` — `ProgramError` enum
- `utils/pda.rs` — PDA seed constants + derivation helpers

**Account validation pattern (native style):**
```rust
// Every instruction handler validates manually — no macros
fn process_place_order(program_id: &Pubkey, accounts: &[AccountInfo], args: PlaceOrderArgs) -> ProgramResult {
    let [market_ai, user_position_ai, order_ai, user_ai, system_program_ai] = accounts else {
        return Err(ProgramError::NotEnoughAccountKeys);
    };
    // ownership checks
    assert_eq!(market_ai.owner, program_id);
    assert_eq!(user_position_ai.owner, program_id);
    // signer checks
    if !user_ai.is_signer { return Err(ProgramError::MissingRequiredSignature); }
    // PDA derivation checks
    let (expected_pda, _bump) = Pubkey::find_program_address(
        &[b"order", market_ai.key.as_ref(), user_ai.key.as_ref(), &args.nonce.to_le_bytes()],
        program_id,
    );
    assert_eq!(order_ai.key, &expected_pda);
    // ...
}
```

**State accounts:**
- `Market` — one per question. Holds vault, resolution state, admin.
- `Order` — one per open order. Stores side, price, size, fill amount.
- `UserPosition` — one per (user, market). Tracks internal balances and open order list.

**Instructions (dispatched via 1-byte discriminant):**

| # | Variant | Caller | Description |
|---|---|---|---|
| 0 | `CreateMarket` | admin | create Market PDA, init USDC vault ATA |
| 1 | `Split` | user | deposit USDC → vault; credit `yes_balance` + `no_balance` |
| 2 | `Merge` | user | debit `yes_balance` + `no_balance`; return USDC from vault |
| 3 | `PlaceOrder` | user | create Order PDA; move balance into `locked_*` |
| 4 | `CancelOrder` | user | close Order PDA; release `locked_*` |
| 5 | `FillOrder` | keeper | validate crossing; swap balances between two UserPositions |
| 6 | `ResolveMarket` | admin | set `resolved = true`, `winning_outcome` |
| 7 | `Redeem` | user | debit winning balance; transfer USDC from vault |
| 8 | `TokenizePosition` | user (opt-in) | mint real SPL YES/NO tokens (user pays ATA cost) |

#### 2. DLOB Keeper (`keeper/`)

A TypeScript daemon that provides decentralized order matching without a central server.

**Flow:**
1. `OrderSubscriber` — `getProgramAccounts` with `memcmp` filter on account discriminant byte + market pubkey; subscribes to changes via WebSocket.
2. `DLOB` — maintains two sorted lists per market: bids (price DESC, FIFO), asks (price ASC, FIFO).
3. Crossing detection — when `best_bid.price >= best_ask.price`, a fill is possible.
4. `Filler` — constructs and submits a `FillOrder` transaction. Permissionless; first keeper to land wins. Earns `fill_fee_bps` of fill size.

#### 3. SDK (`sdk/`)

Hand-written TypeScript client. No IDL auto-generation — types mirror the Borsh layout defined in Rust structs exactly.

- `instructions.ts` — builds `TransactionInstruction` for each of the 9 instructions
- `accounts.ts` — deserializes `Market`, `Order`, `UserPosition` account data using `borsh` npm package
- `pda.ts` — PDA derivation matching the on-chain seeds
- `dlob/` — `DLOB`, `DLOBNode`, `OrderSubscriber`

#### 4. Frontend (`app/`)

Next.js app using `@solana/wallet-adapter` and the SDK.

## Token Model

### Why not plain SPL tokens?

Each SPL Associated Token Account (ATA) costs **~0.00204 SOL** in rent (165 bytes, rent-exempt). A naïve design would require:

| Account | Who pays | When |
|---|---|---|
| YES ATA per user | user | first trade |
| NO ATA per user | user | first trade |
| escrow ATA per resting order | user | each order placed |
| USDC ATA | user | once ever |

That is **~0.006–0.01 SOL before placing a single trade**, multiplied by every market a user participates in.

### Internal-balance model (adopted)

Balances are stored as `u64` fields inside `UserPosition` (a program-owned PDA). SPL tokens are only minted on explicit opt-in via `TokenizePosition`.

```
Default path (zero per-user ATA cost):

  User deposits 100 USDC
       │
       ▼
  transfer USDC → Market vault (one USDC ATA, created once by admin)
  UserPosition.yes_balance += 100
  UserPosition.no_balance  += 100

  All trades update numbers in UserPosition — no ATA overhead.

  After resolution (YES wins):
  redeem(100) → transfer 100 USDC from vault to user

Optional tokenization path (user pays ~0.004 SOL, once, opt-in only):

  TokenizePosition(amount)
       ├─ lazily creates YES/NO mints (if not yet created for this market)
       ├─ creates user YES ATA + NO ATA
       └─ mints real SPL tokens → composable with LP pools, lending, bridges
```

**ATA cost summary:**

| Account | Count | Payer |
|---|---|---|
| Market USDC vault ATA | 1 per market | admin at market creation |
| User USDC ATA | 1 per user lifetime | user (likely already exists) |
| YES/NO ATAs | 0 default; 2 if `TokenizePosition` called | user, opt-in |
| Order escrow ATA | 0 — locked amounts are fields in Order account | n/a |

### On-chain state structs

```rust
// state/market.rs
#[derive(BorshSerialize, BorshDeserialize)]
pub struct Market {
    pub discriminant:     u8,         // = 0, used for getProgramAccounts filter
    pub question_hash:    [u8; 32],
    pub vault:            Pubkey,     // USDC ATA owned by Market PDA
    pub collateral_mint:  Pubkey,     // mock USDC mint on devnet
    pub yes_mint:         Pubkey,     // Pubkey::default() until TokenizePosition called
    pub no_mint:          Pubkey,
    pub end_time:         i64,
    pub resolved:         bool,
    pub winning_outcome:  u8,         // 0 = unresolved, 1 = YES, 2 = NO
    pub admin:            Pubkey,
    pub order_count:      u64,
    pub bump:             u8,
}

// state/order.rs
#[derive(BorshSerialize, BorshDeserialize)]
pub struct Order {
    pub discriminant:  u8,           // = 1
    pub market:        Pubkey,
    pub user:          Pubkey,
    pub side:          u8,           // 0 = bid (buy YES), 1 = ask (sell YES)
    pub price:         u64,          // basis points 0–10000
    pub size:          u64,          // collateral units (6 decimals for USDC)
    pub fill_amount:   u64,
    pub nonce:         u64,
    pub created_at:    i64,
    pub bump:          u8,
}

// state/user_position.rs
#[derive(BorshSerialize, BorshDeserialize)]
pub struct UserPosition {
    pub discriminant:      u8,       // = 2
    pub market:            Pubkey,
    pub user:              Pubkey,
    pub yes_balance:       u64,
    pub no_balance:        u64,
    pub locked_yes:        u64,      // YES locked in open ask orders
    pub locked_no:         u64,      // NO locked in open no-side orders
    pub locked_collateral: u64,      // USDC locked in open bid orders
    pub open_orders:       [Pubkey; 32],
    pub open_order_count:  u8,
    pub bump:              u8,
}
```

## Order Lifecycle

```
User places order
      │
      ▼
PlaceOrder instruction
├─ create Order PDA [b"order", market, user, nonce.to_le_bytes()]
├─ allocate + init Order account via system_program::create_account
├─ bid: locked_collateral += size
   ask: locked_yes += size
└─ append Order.key to UserPosition.open_orders[]

Keeper picks up Order account change (WebSocket)
      │
      ▼
DLOB updated in memory
      │
      ▼ (if best_bid.price >= best_ask.price)
FillOrder instruction (keeper submits)
├─ verify bid.market == ask.market
├─ verify bid.price >= ask.price
├─ compute fill_size = min(bid.remaining, ask.remaining)
├─ bid UserPosition: locked_collateral -= fill_cost; yes_balance += fill_size
├─ ask UserPosition: locked_yes -= fill_size; yes_balance = 0 (sold)
│                    collateral_balance += fill_proceeds
├─ credit keeper: collateral_balance += fill_fee
├─ update Order.fill_amount on both orders
└─ close fully-filled Order accounts (lamports → user)

User sees updated UserPosition balances
```

**1.3 Security constraints (enforced manually per handler)**
- All handlers: verify `account.owner == program_id` for program-owned accounts
- All handlers: verify PDA derivation matches expected seeds
- `place_order`: price must be 1–9999; `end_time` not past; market not resolved
- `fill_order`: bid.side=0, ask.side=1; same market; bid.price >= ask.price; neither order over-filled; keeper is signer
- `resolve_market`: `user_ai.key == market.admin`; `!market.resolved`
- `cancel_order`: `user_ai.key == order.user`; order exists
- `split`/`merge`/`redeem`: amounts > 0; no overflow on balances

**1.4 Rust tests (`solana-program-test`)**
- Full happy path: create → split → place bid + ask → fill → resolve → redeem
- Partial fill: two orders where ask.size > bid.size
- Cancel after partial fill: remaining locked balance released correctly
- Over-fill attempt: `FillOrder` with fill_size > remaining → error
- Double resolve: second `ResolveMarket` → error

### Phase 2: TypeScript SDK (Day 5)

- No IDL — write `accounts.ts` types by hand matching Borsh layout byte-for-byte
- Use `borsh` npm package (`^2`) for `BorshSchema` deserialization
- `instructions.ts`: each function returns a `TransactionInstruction` with manually encoded data:
  ```ts
  // First byte = discriminant, rest = Borsh-serialized args
  const data = Buffer.alloc(1 + argSize);
  data.writeUInt8(3, 0); // PlaceOrder discriminant
  borsh.serialize(PlaceOrderArgsSchema, args, data.slice(1));
  ```
- `pda.ts`: `PublicKey.findProgramAddressSync` with same seeds as on-chain
- `OrderSubscriber`: `getProgramAccounts` with two `memcmp` filters:
  - offset 0, bytes `[1]` (Order discriminant)
  - offset 1, bytes `market.toBase58()` (market pubkey at offset 1)

### Phase 3: Keeper Bot (Day 6)

- `OrderSubscriber.subscribe()` → builds initial DLOB, then updates on account changes
- Crossing check: called on every DLOB mutation
- `Filler.fill(bid, ask)`:
  1. Simulate with `connection.simulateTransaction` → skip if error
  2. Send with `sendAndConfirmTransaction`
  3. On `AccountNotFound`: order was filled by another keeper, remove from DLOB
  4. On success: update fill amounts in local DLOB cache
- Fee: `fill_fee_bps = 5` (0.05% of fill size), paid out inside `FillOrder` handler

## Differences from Drift DLOB

Drift's DLOB includes a JIT auction, AMM fallback, vAMM oracle, and perpetual futures. This project is a simplified DLOB scoped to binary prediction markets:

- **No JIT auction** — orders fill directly from the book
- **No AMM fallback** — orders rest until a crossing counterpart appears or cancelled
- **No oracle price peg** — prices are user-set in basis points (0–10000 = 0%–100%)
- **Binary markets only** — YES/NO outcomes; no perpetuals or multi-outcome

## Open Questions / Decisions Deferred

1. **Oracle for resolution** — Phase 1 uses an admin keypair. Phase 2 can integrate Switchboard VRF or Pyth price feeds for automated resolution.
2. **Keeper incentives** — Flat `fill_fee_bps` for now. Could become a priority-fee auction among competing keepers.
3. **Multi-outcome markets** — Binary only. N-outcome extension needs N internal balance fields or a dynamic vec.
4. **GTD order expiry** — Keeper could submit `CancelOrder` for expired orders and earn a cleanup fee.
5. **Price precision** — u64 basis points (0–10000, 2 decimal places). Could switch to u64 with 6 decimals if tighter pricing is needed.
6. **Front-running** — Keepers can race on fills. A short JIT window (like Drift's) or commit-reveal scheme could mitigate if needed.
